Architectural Deep Dive: Replicating Chub AI Prompt Structures and Character JSON Parsing for GalaxyOS ApplicationsThe ecosystem of personalized artificial intelligence has undergone a rapid and profound evolution, pivoting away from monolithic, single-purpose conversational agents toward decentralized, highly customizable persona engines. Central to this paradigm shift are platforms such as Chub AI, SillyTavern, and Agnai, which rely on standardized character definition files to instantiate complex, persistent personas across varied Large Language Models (LLMs). The modern application landscape dictates that these robust AI character frameworks no longer exist merely as isolated web interfaces. Instead, they must function as embedded, system-level integrations within advanced mobile operating environments.This report provides an exhaustive technical blueprint for constructing a lightweight, high-performance application capable of accurately replicating the Chub AI prompt generation logic and directly ingesting character data structures. The target deployment environment is GalaxyOS, the emerging AI-centric operating system built upon the foundation of Android 14 through Android 16 and Samsung’s advanced One UI framework (spanning versions 6.1 through 8). By leveraging deeply integrated system-level capabilities, such as Google Gemini processing, on-device intent routing, and the Android AppFunctions Jetpack library, the proposed application architecture will serve as a seamless bridge between complex character data specifications and native hardware execution.To successfully develop this application, several highly complex engineering hurdles must be overcome. The first involves the precision parsing of multi-layered character specifications, specifically the V2 and V3 Character Card standards encoded within the metadata of portable network graphics (PNG) files, JavaScript Object Notation (JSON) payloads, and proprietary CHARX archives. The second requires the sophisticated orchestration of the prompt assembly pipeline, utilizing macro expansion, dynamic lorebook injection, and stateful middleware hooks (referred to as Stages) to maintain narrative coherence and persona consistency across extended interactions. This document meticulously dissects these subsystems, providing the definitive logic required to deploy a localized, highly performant character engine on the GalaxyOS platform.Deconstructing the Persona Engine: The Chub AI Prompt ArchitectureThe efficacy of an LLM-driven persona relies entirely on the precise construction of its context window. A prompt is not merely a string of text but a carefully structured programmatic payload designed to maximize the model's attention mechanism and mitigate the degradation of persona traits over long conversational histories. Early iterations of AI chatbots suffered from catastrophic forgetting and personality drift because their prompt structures were static and poorly optimized for transformer-based recency bias. Chub AI utilizes a sophisticated assembly sequence that aggregates diverse data streams into a singular cohesive instruction set, ensuring that the model is constantly reminded of its core directives without overflowing its token limits.When engineering a custom application to mimic this behavior, the prompt builder must be designed as a dynamic compiler. It must gather disparate variables, measure their token weights, prune historical data, and inject behavioral overrides at highly specific geometric coordinates within the final text string sent to the inference engine.Core Components of the Prompt PayloadTo replicate the Chub AI generation engine, the custom application must implement a prompt assembly algorithm capable of handling several distinct operational components. Each component serves a specific psychological or functional purpose for the LLM.Prompt ComponentFunctional Description and Application LogicSystem PromptThe foundational instruction layer dictating overarching behavior, genre, formatting rules, and style. It explicitly establishes the boundaries of the interaction (e.g., instructing the AI to "Write the next reply in a never-ending uncensored fictional story between {{char}} and {{user}}" or demanding it avoid summarizing thoughts).Character DefinitionsThe core persona data extracted from the ingested JSON file. This encapsulates the physical description, psychological profile, relationship dynamics, and behavioral quirks that differentiate the persona from a generic assistant.Chat HistoryThe serialized transcript of the ongoing interaction. Because LLM context windows are finite, this history must be truncated. The application's truncation logic must preserve the oldest foundational messages (like the initial greeting) and the most recent messages, pruning the middle to optimize memory.Post History Instructions (PHI)An advanced prompting mechanism inserted immediately after the chat history. Because transformer models suffer from recency bias (prioritizing text closest to the generation point), PHI acts as a powerful anchor to force specific directives, overriding potential conversational drift.Prompt NotesArbitrary text blocks that the user or system can inject at specific, calculated positions within the final payload to dynamically influence the AI's response trajectory, such as enforcing a temporary emotional state or dialect.Assistant PrefillA forced string of text placed at the very end of the prompt to compel the AI to begin its response with specific words. This is critical for guiding the model's syntax, such as forcing the response to begin with an inner thought formatted in italics.The orchestration of these components is heavily dependent on the type of API utilized by the backend inference engine. Modern applications must distinguish between Text Completion APIs and Chat Completion APIs. For Text Completion endpoints (such as base Llama models running on local hardware), the application must concatenate the entire payload into a single raw text string. Conversely, Chat Completion APIs require the prompt to be formatted as an array of JSON message objects, categorized by roles such as system, user, and assistant. The custom application must possess a modular formatting layer capable of translating the assembled components into either format immediately prior to transmission.Furthermore, advanced frontends like SillyTavern demonstrate that robust prompt builders must allow users to override default templates. For Text Completion APIs, the application should support an automated template derivation system. This functions by comparing a cryptographic hash of the chat template defined within a model's tokenizer_config.json file against a repository of default application templates, automatically selecting the optimal formatting structure for the user's specific LLM. If a user wishes to manually override these settings, the application must provide mechanisms to edit Context Templates, Instruct Templates, and custom stopping strings, while offering a fallback to restore default configurations by clearing internal cache logs (such as a content.log file tracking modified defaults).Algorithmic Implementation of Prompt Formatting ParadigmsChub AI processes the assembly of these prompt components using two primary architectural paradigms. Both methodologies must be supported by the native GalaxyOS application to ensure total cross-compatibility with imported character cards and user expectations.The Normal Formatting SequenceIn the standard "Normal" configuration, the prompt builder concatenates the components in a hardcoded, rigid sequence. This approach is deterministic and requires minimal computational overhead during the compilation phase. The logic flow operates strictly top-to-bottom, ensuring that the model processes foundational rules before encountering specific narrative context.The application's compilation algorithm for Normal formatting must execute the following sequence: First, the System Prompt establishes the absolute rules of the engagement. Second, the Character Definitions introduce the actors, their personalities, and the overarching scenario. Third, the Chat History provides the immediate situational context, feeding the model the recent dialogue. Fourth, the Post History Instructions supply the immediate behavioral anchor, reminding the model of formatting rules just before it begins generation. Fifth, the Prompt Note is appended, depending on its dynamically assigned position within the configuration settings. Finally, the Assistant Prefill sets the starting token for generation, seamlessly guiding the LLM into the required syntactic structure.While reliable and easy to implement, this fixed structure limits the ability to intertwine system directives with character data. If a model requires personality definitions to be stated before the overarching system rules to maintain focus, the Normal formatting sequence cannot accommodate this inversion.The Macro (AGNAI) Formatting SequenceTo overcome the limitations of rigid concatenation, the "AGNAI" or Macro formatting engine represents a paradigm shift toward highly templated, dynamic prompt generation. Instead of a fixed sequence of blocks, the system prompt itself acts as a master template containing variable injection points. The custom application must deploy a highly optimized text-replacement engine—conceptually similar to Mustache or Handlebars templating—to parse the prompt string and inject the appropriate data blocks prior to inference.The application's compiler must support a comprehensive suite of standard macros to ensure compatibility with Chub AI cards.Macro SyntaxData Injection Source and Application Logic{{personality}}Injects the character's core traits, extracting the text directly from the personality field of the ingested JSON schema.{{scenario}}Injects the environmental context or current plot setup, extracted from the scenario field.{{memory}}Injects dynamic background information triggered from the external Lorebook subsystem. This data fluctuates based on keyword scanning algorithms executed on the chat history.{{example_dialogue}}Injects few-shot prompting examples to calibrate the LLM's syntactical tone, drawn from the mes_example field.{{summary}}Injects rolling conversation summaries. These summaries are generated asynchronously by the AI to compress older chat history, allowing long-term memory persistence beyond the hard token limit of the context window.{{profile}}Injects the human user's personal definitions and physical descriptions, ensuring the character can reference the user accurately.{{char}} / <char>Dynamic name replacement for the AI persona. The application must replace this with the string located in the name or V3 nickname fields.{{user}} / <user>Dynamic name replacement representing the human participant.By implementing this macro engine, a user can construct a prompt string such as "{{personality}} {{memory}} {{example_dialogue}} Write {{char}}'s next reply in an immersive roleplay between {{char}} and {{user}}.". The application dynamically reorders the payload based entirely on this template, offering precise control over the model's attention mechanism. The backend logic requires a multi-pass regular expression scanner to identify macro brackets, extract the corresponding data from the application's internal state management system, and replace the substrings while continuously monitoring the overall token count to prevent exceeding the model's maximum context length.Middleware and Pipeline Manipulation: The Stage ArchitectureTo construct a highly advanced character application that rivals established web frontends, static prompt generation is insufficient. Chub AI implements a powerful middleware concept known as "Stages". Stages are, at their core, lightweight, React-based websites or functional scripts that operate within the context of a chat. They intercept and modify the data flow between the user interface and the LLM execution layer, allowing for dynamic narrative interventions, environmental simulations, and complex state tracking without altering the base character file.To achieve feature parity, the new GalaxyOS application must construct a similar event-driven middleware pipeline, exposing a programmable interface to developers or power users.Top-Down Lifecycle HooksThe Stage middleware operates on highly specific lifecycle events, injecting programmatic logic directly into the conversational loop. The architectural blueprint for the custom application requires the implementation of two primary synchronization points: beforePrompt and afterResponse.When the user submits a message, but before the final payload is compiled and transmitted to the local inference engine or external API, the application must trigger the beforePrompt function. During this phase, the middleware operates with elevated privileges over the prompt string. It can mutate the user's message, automatically appending or altering user text to correct formatting, apply translations, or enforce specific narrative constraints. Furthermore, the beforePrompt hook can append hidden directives to the system prompt based on internal logic. Crucially, this stage allows for the attachment of "system messages." These are user-interface elements that render visible text to the human user (such as a notification that a character's simulated affection level has increased) but are explicitly stripped from the LLM's context payload. This isolation prevents the LLM from becoming confused by out-of-character notifications or attempting to hallucinate its own logic blocks in subsequent replies. Finally, the beforePrompt stage is responsible for saving the dynamic internal state of the conversation, persisting variables like inventory systems, time-of-day trackers, or emotional meters to the local database.Once the inference engine completes its generation and the serialized token stream is returned, the application must execute the afterResponse hook. This middleware processes the incoming data before it is rendered to the UI. The capabilities at this stage include response mutation, allowing the application to parse the LLM output to remove unwanted formatting, correct hallucinated asterisks, or strictly enforce a specific response length constraint before the user ever sees the text. Similar to the pre-prompt stage, afterResponse can attach human-readable commentary or status updates alongside the bot's reply and execute state updates, altering internal variables based on actions the LLM described in its generation.Experimental Bottom-Up ServicesIn addition to intercepting text during the conversational loop, advanced implementations of the Stage interface permit asynchronous "bottom-up" API calls. A comprehensive recreation of the Chub engine within a native OS environment should scaffold support for multimedia generation services triggered autonomously by the prompt pipeline.According to the TypeScript interface structures underlying Chub Stages, the application architecture should provision endpoints for several experimental services. The transcribe service handles requests for speech-to-text conversion from audio inputs, transforming voice messages into standard text before the beforePrompt hook fires. The makeVoice service handles requests to create a voice clone based on provided audio, operating asynchronously to generate text-to-speech outputs utilizing the LLM's response. Additionally, the modelGen service represents a highly experimental feature to request a 3D model generation based on a text prompt or image reference, with optional support for textures and rigging. Finally, a dedicated textGen service allows the stage itself to request separate text generation from the LLM, independent of the main conversational loop, enabling background processing tasks like summarizing hidden memories or calculating hidden character motives. While some of these features remain experimental or pending within original implementations, providing interfaces for them ensures the custom application is future-proofed for multimodal, agentic workflows expected on modern operating systems.Character File Specifications and Data Ingestion SchemaThe core objective of the application is the seamless ingestion and instantiation of character files downloaded from major community repositories like Chub AI, CharacterHub, or SillyTavern. These files are not monolithic blocks of text; they adhere to meticulously defined, hierarchical structural specifications. To guarantee total interoperability and prevent parsing failures, the application's data layer must strictly adhere to the V1, V2, and V3 character card standards, employing robust fallback and backfilling mechanisms.Legacy Validation: The V1 SpecificationThe V1 standard is the foundational schema adopted by early platforms such as the Pygmalion booru and CharacterHub. While largely superseded, a vast archive of legacy characters relies on this format. A V1 parser must extract top-level string properties directly from the root of the JSON object. The application must map the name string representing the character's identity, the description string serving as the primary bulk text defining the persona, and the personality string utilized for discrete psychological traits. Furthermore, it must extract the scenario string establishing the situational context, the first_mes string dictating the greeting message presented upon initiating a chat, and the mes_example string containing sample dialogues utilized for few-shot learning.Structural Evolution: Character Card V2 SpecificationThe V2 specification was engineered to embed deeper programmatic control directly within the character file, addressing the loss of persona coherence caused by community "jailbreaks" and uncontrolled, highly variable global system prompts. Structurally, V2 maintains backward compatibility by nesting all new programmatic properties inside a dedicated root data object.The application’s JSON parser must validate a V2 object by ensuring the spec field strictly equals "chara_card_v2" and the spec_version strictly equals "2.0". Beyond the foundational V1 fields embedded within the data object, the parser must extract and accurately map numerous critical V2 extensions.V2 Field SpecificationData StructureApplication Parsing and Execution Logiccreator_notesStringMeta-information provided by the author. The prompt builder must actively quarantine this field; it must never be injected into the LLM prompt sequence to prevent token waste and hallucination.system_promptStringCharacter-specific foundational instructions. The application must execute logic that overrides global user settings if this field is populated, utilizing the card's specific system prompt as the primary directive.post_history_instructionsStringInstructions scheduled for injection immediately before the assistant prefill. Essential for enforcing behavioral rules in long chats, circumventing recency bias.alternate_greetingsArray of StringsA collection allowing the user to select varied opening scenarios. CCv3 operates as a superset of V2, meaning the application must support all previous properties while implementing robust handling for new fields. The parser identifies a V3 file by validating spec as "chara_card_v3" and parsing spec_version as a float to dynamically handle subsequent minor version updates (e.g., version 3.1).The application must implement ingestion logic for several highly specific V3 advancements. The nickname field introduces a discrete string that, if populated, commands the application's prompt generator to replace all instances of the {{char}}, <char>, and <bot> macros with this string instead of the standard name property. The group_only_greetings field introduces an array of strings dedicated to multi-character simulated group chats. The application's UI logic must dynamically filter available greetings based on whether the current chat architecture involves a single bot or a grouped environment.Furthermore, V3 standardizes temporal tracking via the creation_date and modification_date fields, encoded as Unix timestamps in UTC. The specification mandates strict UI behavior: applications must not allow users to arbitrarily edit these dates, and should default the variables to 0 if unknown, ensuring privacy and reliable version tracking. Localization is supported via the creator_notes_multilingual field, a record mapping standardized language codes (e.g., en, es, jp) to localized notes. The parser logic must dictate that if this object is present but missing an en key, the standard creator_notes string acts as the English fallback. Finally, V3 introduces the assets array, containing objects that define multimedia assets via type, uri, name, and ext properties, laying the groundwork for multimodal interaction.The Decryption and Extraction Pipeline: PNG and CHARX IngestionWhile importing a raw JSON file is programmatically trivial, the cultural standard for sharing AI characters across platforms revolves around Portable Network Graphics (PNG) files. These images serve a highly efficient dual purpose: they act as the visual avatar for the character in the user interface and operate as the physical data carrier for the entire JSON schema using sophisticated steganographic metadata embedding.To achieve direct, drag-and-drop import functionality into the GalaxyOS application, a highly robust extraction pipeline must be constructed to decrypt the embedded payloads from image files without relying on external web APIs.The Steganographic PNG Chunk Parsing AlgorithmContrary to common assumption, character data is not encoded as standard EXIF metadata in modern iterations, but rather within the highly specific tEXt chunks of the PNG binary specification. The algorithmic pipeline for extraction must be carefully engineered to read binary streams directly from local storage.The extraction sequence proceeds as follows: First, the application opens the target PNG file in a raw binary stream. Second, the parser iterates through the hierarchical PNG structure, specifically targeting tEXt chunks while deliberately ignoring standard image data blocks (IDAT) to conserve processing power. Third, the parser evaluates the keyword associated with the tEXt chunk. It must actively search for the ccv3 key (indicating a modern V3 standard) or the legacy chara key (indicating V2). If both keys are present simultaneously—a common occurrence when creators attempt backward compatibility—the application's decision logic must prioritize the ccv3 chunk to ensure the most modern schema is utilized.Once the correct chunk is identified, the application extracts the payload. This payload is not raw JSON, but rather a Base64-encoded string. The application must process this payload through a standard Base64 decoding algorithm, translating it into a raw byte array. This raw byte array is then decoded via UTF-8 formatting into an expansive string, which is finally passed through a JSON parser to instantiate the native object in the application's memory. Python-based implementations of this logic typically utilize the base64 and json standard libraries, executing commands such as json.loads(base64.b64decode(clean_text).decode("utf-8")). Javascript implementations for the frontend operate on similar asynchronous atob decryption paradigms.Error Handling and Legacy BackfillingThe extraction pipeline must be engineered with defensive programming paradigms to handle malformed, corrupted, or maliciously crafted JSON files. Broken JSON outputs or corrupted base64 strings are exceedingly common in community-generated files due to poorly formatted manual edits or errors from lesser frontends. The application should implement recursive exception handling that gracefully defaults missing critical arrays (like alternate_greetings) to empty lists and strips unescaped control characters before attempting deserialization.Furthermore, if the parser successfully detects a chara chunk but no ccv3 chunk, the system must engage an automated "backfilling" protocol. This protocol systematically maps the legacy V2 fields into a newly instantiated V3 object in memory, ensuring the rest of the application's logic only ever has to deal with the V3 schema. However, the V3 specification strictly mandates that applications performing this backfill must render a warning notification to the user, alerting them that legacy data structures are being dynamically upgraded and some V3-specific features may be unavailable.V3 Proprietary Packaging: The CHARX Archive FormatWhile standard JSON and PNG embeddings persist as the norm, the CCv3 specification introduces .charx, a proprietary archive format designed to encapsulate both the JSON schema and external multimedia assets into a single distributable file.To process a CHARX file, the application must deploy a ZIP extraction utility capable of reading archive directories. The expected directory structure locates the primary card.json at the root of the archive, accompanied by a comprehensive assets/ directory. The application must algorithmically map the {type} variable defined in the JSON to specific folders within the archive, routing images to assets/{type}/images/, audio files to assets/{type}/audio/, and custom execution code (such as embedded JavaScript) to assets/{type}/code/. The internal application router must then resolve Uniform Resource Identifiers (URIs) utilizing the specialized embeded:// protocol (e.g., embeded://path/to/asset.png), extracting the necessary binary data directly from the archive and serving it to the UI or execution layer.Dynamic Context Injection: Lorebooks and Character BooksThe static context window of an LLM cannot possibly sustain the vast encyclopedic knowledge required for comprehensive world-building, intricate magic systems, or deep character histories. To solve this critical limitation, Chub AI and similar platforms rely heavily on "Lorebooks"—dynamic, event-driven dictionaries that inject contextual data into the prompt only when specific semantic triggers are activated in the conversation. When a lorebook is embedded directly inside a specific character's JSON file, it is structurally referred to as a "Character Book".The application's parser and prompt generation pipeline must feature a highly optimized, multi-pass text-scanning engine to process these rules seamlessly without adding noticeable latency between the user's input and the LLM's response.Execution Parameters and Token BudgetsThe integration of a lorebook into the generation pipeline requires rigorous computational control over memory management to prevent context window overflow. If too many lorebook entries are triggered simultaneously, the fundamental system prompt and chat history will be truncated, destroying the persona.To mitigate this, the application must implement a variable "Scan Depth." The engine must scan only the most recent N messages of the chat history for matching keywords. Scanning begins at the end of the chat array and moves backward. For example, a scan depth of 2 forces the algorithm to process only the user's latest input and the AI's immediate prior response, ignoring older context.Simultaneously, the engine must enforce a strict "Token Budget"—a hard limit on the total number of tokens the lorebook is permitted to inject per generation cycle. The system must sort all activated entries by their assigned priority value, injecting the highest priority data first. Once the mathematical token limit is reached, all subsequent keyword matches are deliberately ignored, ensuring the prompt remains stable.Furthermore, the application must support "Recursive Scanning," an advanced computational requirement where the text injected by one lorebook entry is subsequently scanned for keywords to trigger secondary lorebook entries. This creates complex, cascading dependencies between world-building variables, requiring the application's scanner to execute in a while loop until no new keys are triggered or the token budget is exhausted.V3 Lorebook Schema Enhancements and DecoratorsThe CCv3 specification severely upgrades the programmatic capability of Character Books, necessitating a highly sophisticated parsing engine. The use_regex field is now a mandatory boolean requirement, forcing the application's keyword scanner to support complex Regular Expressions for pattern matching rather than simple string equivalence. Additionally, the constant boolean flag dictates that, when true, the entry bypasses keyword scanning entirely and is injected permanently into the context window.Perhaps the most complex architectural feature of the V3 Lorebook is the implementation of "Decorators." Decorators are inline syntactical tags injected directly into the content string (the actual text sent to the LLM), prefaced with @@ and terminated by a newline. They allow creators to exert granular control over individual entries without requiring the JSON object schema to expand infinitely.The application’s prompt compiler must parse the text block, extract the decorators, execute their corresponding logic within the prompt assembly sequence, and then critically strip the @@ strings from the final payload to prevent polluting the LLM prompt with programmatic syntax.Decorator SyntaxEngine Execution Logic@@positionDictates the exact geometric insertion point within the prompt array. If the overall context window is reaching its absolute maximum capacity during final prompt compilation, any entry bearing this decorator must be the first to be jettisoned to conserve tokens, regardless of its standard priority score.@@additional_keysConditional logic parameter. The entry will only activate if the primary keyword and the defined additional keys are both present within the scanned chat log.@@exclude_keysNegative conditional logic.
